도메인 모델은 처음부터 완벽하게 맞추기 어렵습니다. 특히 제품 정책이 빠르게 바뀌는 초기 서비스에서는 “지금 맞는 추상화”가 몇 달 뒤에는 오히려 복잡도를 만들기도 합니다.

이벤트 도메인에서 그런 경험이 있었습니다.

## 처음에는 왜 template을 분리했나

초기 이벤트 모델에서는 `event_template`과 `event`를 분리했습니다.

```text
event_template
  - 공통 상품 정의
  - 이름, 설명, 가격, 소요 시간, 대상 유형

event
  - 실제 운영 인스턴스
  - 노출 기간, 상태, 제휴사 조건, 입점업체 연결
```

이렇게 나눈 이유는 명확했습니다. 같은 상품 정의를 여러 이벤트 운영 인스턴스가 공유할 수 있다고 봤기 때문입니다. 공통 정의와 실제 운영 상태를 분리하면 재사용성이 높아지고, 이벤트 운영 정책이 늘어날 때 유연할 것이라고 판단했습니다.

## 정책이 바뀌면 좋은 추상화도 비용이 된다

운영이 진행되면서 이벤트는 template보다 실제 event 자체가 더 중요한 단위가 되었습니다. 가격, 이미지, 노출 대상, 옵션, 전문가 연결 같은 값이 운영 이벤트마다 달라졌고, 입점업체 검수 workflow도 실제 이벤트 단위로 흘러가야 했습니다.

이 상황에서 template 분리는 점점 비용이 됐습니다.

- 조회할 때 조인이 늘어난다.
- 수정 권한과 검수 흐름의 기준이 흐려진다.
- 이미지 FK가 template과 event 사이에서 헷갈린다.
- 이벤트 유형 계산이 여러 테이블에 퍼진다.
- 실제 운영자가 보는 단위와 데이터 모델이 어긋난다.

그래서 `event_template`의 핵심 컬럼을 `event`로 병합했습니다.

## 마이그레이션에서 중요했던 것

이런 구조 개편은 단순히 컬럼을 옮기는 작업이 아닙니다. 기존 데이터가 이미 있고, 이미지나 옵션 같은 연결 데이터도 있기 때문에 데이터 보존과 rollback을 같이 봐야 합니다.

마이그레이션에서 신경 쓴 지점은 다음과 같습니다.

## 1. 핵심 컬럼 이전

template에 있던 이름, 설명, 가격, 소요 시간, VAT 포함 여부, 이벤트 대상 유형 같은 값을 event로 옮겼습니다.

```text
event_template.name        -> event.name
event_template.description -> event.description
event_template.price       -> event.price
event_template.duration    -> event.duration
```

이렇게 하면 운영 event 하나만 봐도 실제 서비스에 필요한 핵심 정보를 알 수 있습니다.

## 2. 이벤트 대상 유형 계산

기존 데이터에서는 고객 유형이나 제휴사 존재 여부를 조합해 이벤트 대상 유형을 계산해야 했습니다. migration 안에서 기존 데이터를 읽고 새로운 enum 값으로 매핑했습니다.

이 단계에서 중요한 것은 애플리케이션 코드의 현재 로직에만 의존하지 않는 것입니다. migration은 독립적으로 재실행 가능한 데이터 변환이어야 하므로, DB에 남아 있는 기존 값만으로 새 컬럼을 채울 수 있어야 합니다.

## 3. 이미지 FK 보존

이미지 테이블이 template 기준으로 연결되어 있었다면, 병합 이후에는 실제 event 기준으로 연결되어야 합니다. FK를 단순 삭제 후 재생성하면 데이터가 끊길 수 있으므로, 기존 template ID와 event ID 매핑을 보존한 상태에서 FK를 이동해야 합니다.

```text
event_image.event_template_id
  -> event_image.event_id
```

이 과정은 운영 데이터 보존의 핵심이었습니다.

## 4. rollback 고려

구조 개편 migration은 rollback 가능성을 같이 봐야 합니다. 물론 모든 도메인 migration을 완벽하게 되돌릴 수는 없지만, 적어도 다음을 고려해야 합니다.

- 기존 컬럼 삭제 전 중간 상태 보존
- FK 이동 순서
- nullable 전환과 backfill 순서
- 새 컬럼 검증 후 old table/column 제거
- 장애 발생 시 되돌릴 수 있는 rollback script

특히 운영 데이터가 있는 서비스에서는 “한 번에 깨끗하게 바꾼다”보다 “중간 상태에서도 데이터가 안전하다”가 더 중요합니다.

## 배운 점

처음부터 event 중심 모델을 만들었다면 더 단순했을 수도 있습니다. 하지만 당시에는 template 분리가 합리적인 판단이었습니다. 중요한 것은 추상화가 틀렸다고 부끄러워하는 것이 아니라, 정책이 바뀌었을 때 데이터 보존과 운영 안정성을 지키면서 구조를 바꿀 수 있는가입니다.

도메인 모델링은 한 번의 정답이 아니라, 서비스가 학습한 내용을 데이터 구조에 다시 반영하는 과정에 가깝습니다.
